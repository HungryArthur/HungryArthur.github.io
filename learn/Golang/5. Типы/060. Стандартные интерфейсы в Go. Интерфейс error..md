Интерфейсы в Go играют ключевую роль в обеспечении гибкости и модульности кода. В стандартной библиотеке Go широко используются интерфейсы для создания универсальных решений, таких как работа с потоками данных, обработка ошибок и сериализация. В этом разделе мы рассмотрим стандартный интерфейс `error`, который, как мы обсуждали ранее, является интерфейсом, а также другие важные интерфейсы стандартной библиотеки, такие как `io.Reader`, `io.Writer` и `Stringer`. Мы начнём с подробного анализа `error`, включая создание пользовательского типа ошибки, и затем рассмотрим другие простые, но мощные интерфейсы.

#### Интерфейс error
Ранее мы обсуждали тип `error` как способ обработки ошибок в Go (см. [предыдущий урок](https://stepik.org/lesson/1980241/step/1?unit=2008067)). На самом деле `error` — это встроенный интерфейс, определённый в стандартной библиотеке. Он прост и содержит всего один метод:

```go
type error interface {
	Error() string
}
```

Любой тип, реализующий метод `Error() string`, автоматически соответствует интерфейсу `error`. Это позволяет создавать пользовательские типы ошибок, добавляя дополнительную информацию, например, код ошибки или контекст, сохраняя совместимость с механизмами обработки ошибок в Go.

**Пример с пользовательским типом ошибки:**
Рассмотрим сценарий, где система обработки заказов возвращает пользовательскую ошибку с кодом и описанием.
```go
package main

import "fmt"

type OrderError struct {
	Code    int
	Message string
}

func (e OrderError) Error() string {
	return fmt.Sprintf("Order error %d: %s", e.Code, e.Message)
}

type Order struct {
	ID string
}

func processOrder(o Order) error {
	if o.ID == "" {
		return OrderError{Code: 400, Message: "empty order ID"}
	}
	return nil
}

func main() {
	defer fmt.Println("Processing completed")
	
	order := Order{ID: ""}
	err := processOrder(order)
	if err != nil {
		fmt.Println("Error:", err.Error())
		return
	}
	fmt.Println("Order processed successfully")
}
```
**Вывод:**
```vbnet
Error: Order error 400: empty order ID
Processing completed
```

**Пояснение:**
Структура `OrderError` реализует интерфейс `error` через метод `Error`, возвращающий форматированное сообщение об ошибке. Функция `processOrder` возвращает `OrderError` при пустом `ID`. В `main` ошибка обрабатывается стандартным способом, а `defer` выводит `"Processing completed"`. Это показывает, как пользовательские ошибки интегрируются с интерфейсом `error`.
