#### Зачем вообще нужен context?
Когда мы пишем программы с горутинами, часто бывает так:
- Одна горутина что-то долго считает
- Другая горутина решает, что результат уже не нужен
- Нужно быстро и безопасно прервать первую горутину

Или ещё проще — хочется передать между горутинами небольшую информацию (например, ID задачи), но не хочется создавать отдельные каналы или глобальные переменные.

Пакет `context` решает эти две задачи единым и красивым способом.

#### Главные методы любого контекста
У любого контекста есть четыре ключевых метода:
- `Done() <-chan struct{}` — возвращает канал, который закрывается, когда контекст отменён. Это основной способ «слушать» отмену.
- `Err() error` — возвращает причину отмены (например, `context.Canceled` или `context.DeadlineExceeded`). Вызывается только после закрытия `Done()`.
- `Deadline() (time.Time, bool)` — возвращает дедлайн (если есть) и флаг, установлен ли он вообще.
- `Value(key any) any` — возвращает значение, прикреплённое через `WithValue`.

Эти методы есть у всех контекстов — и у корневых, и у дочерних.

В этом шаге мы рассмотрим три самых простых контекста.

#### 1. context.Background() — «пустой» корневой контекст
Это начало всей цепочки и самый базовый контекст. Он:
- Никогда не отменяется
- Не содержит данных
- Не имеет дедлайна

Используется только в качестве отправной точки и не содержит никакой логики.
```go
package main

import (
	"context"
	"fmt"
)

func main() {
	ctx := context.Background()

	fmt.Printf("Тип: %T\n", ctx)
	fmt.Println("Done() возвращает канал:", ctx.Done() != nil)
	fmt.Println("Err():", ctx.Err())
	deadlineTime, ok := ctx.Deadline()
	fmt.Println("Deadline():", deadlineTime, ok)
}
```
**Вывод:**
```csharp
Тип: context.backgroundCtx
Done() возвращает канал: false
Err(): <nil>
Deadline(): 0001-01-01 00:00:00 +0000 UTC false
```

#### 2. context.TODO() — «я пока не знаю, какой контекст нужен»
Полностью аналогичен `Background()`, но с другим смыслом:
- Вы пишете код и понимаете, что нужен контекст
- Но пока не решили — с отменой, таймаутом или просто пустой
- Ставите `context.TODO()` как заглушку

Это сигнал для вас и коллег: «тут надо будет потом заменить на правильный контекст».
```go
package main

import (
	"context"
	"fmt"
)

func worker(ctx context.Context) {
	fmt.Println("Работаю с контекстом...")
	// потом сюда добавим какую то логику...
}

func main() {
	worker(context.TODO()) // пока заглушка
}
```

#### Контексты можно вкладывать друг в друга — как дерево
Самое важное свойство контекстов — они образуют **дерево**:
- Есть один корневой контекст
- Из него создаются дочерние контексты
- Из дочерних — внучатые, и так далее
- Если родитель отменяется — все его дети отменяются автоматически

Это как дерево: отмена «протекает» вниз по всем веткам. 

#### Пример дерева контекстов
```css
package main

import (
	"context"
	"fmt"
)

func main() {
	root := context.Background()

	child1 := context.WithValue(root, "level1", 1)
	child2 := context.WithValue(child1, "level2", 2)
	child3 := context.WithValue(child2, "level3", 3)
	child4 := context.WithValue(child2, "level4", 4)
	_ = child4

	fmt.Println("child2 видит данные из child1?", child2.Value("level1") != nil)
	fmt.Println("Данные из child1: ", child2.Value("level1"))
	fmt.Println("child2 видит данные из child2?", child2.Value("level2") != nil)
	fmt.Println("Данные из child2: ", child2.Value("level2"))
	fmt.Println("child3 видит данные из child4?", child3.Value("level4") != nil)
	fmt.Println("child3 видит данные из child1?", child2.Value("level1") != nil)
	fmt.Println("Данные из child1: ", child3.Value("level1"))
}
```
**Вывод:**
```yaml
child2 видит данные из child1? true
Данные из child1:  1
child2 видит данные из child2? true
Данные из child2:  2
child3 видит данные из child4? false
child3 видит данные из child1? true
Данные из child1:  1
```

Данный пример наглядно демонстрирует иерархическую природу контекстов в Go, которая реализуется в виде дерева: корневой контекст `root` создается через `context.Background()`, а каждый последующий контекст (`child1`, `child2` и т.д.) порождается от родительского с добавлением своей пары "ключ-значение". Как видно из вывода, дочерний контекст имеет доступ к значениям, сохраненным во всех его предках (например, `child3` видит значение `"level1"` из `child1`), но не имеет доступа к значениям "братских" или "двоюродных" контекстов, созданных от того же родителя (например, `child3` не видит `"level4"` из `child4`, так как оба они происходят от `child2`, но находятся на разных ветках дерева). Эта структура обеспечивает контролируемое распространение данных по цепочке вызовов, одновременно изолируя параллельные ветки выполнения.

#### 3. context.WithValue() — передать данные между горутинами
Самый простой способ передать информацию «вниз по дереву» — использовать `WithValue`. Он создаёт дочерний контекст и прикрепляет к нему значение. Для создания контекста со значением требуется передать уже существующий контекст.
```go
package main

import (
    "context"
    "fmt"
)

func main() {
    ctx := context.Background()
    
    ctx = context.WithValue(ctx, "userID", 777)
    ctx = context.WithValue(ctx, "debug", true)
    
    fmt.Println("userID =", ctx.Value("userID"))
    fmt.Println("debug  =", ctx.Value("debug"))
    fmt.Println("нет ключа =", ctx.Value("requestID")) // nil
}
```
**Вывод:**
```lua
userID = 777
debug  = true
нет ключа = <nil>
```

#### Ключи в WithValue — важный нюанс
Чтобы избежать конфликтов, ключи делают непереиспользуемыми типами:
```go
type userIDKey string

const userKey userIDKey = "userId"

ctx = context.WithValue(ctx, userKey, 777)
value := ctx.Value(userKey).(int) // безопасно!
```

Контекст принято передавать первым параметром и называть `ctx context.Context`.  
Никогда не сохраняйте контекст в структуре — только передавайте его дальше.
