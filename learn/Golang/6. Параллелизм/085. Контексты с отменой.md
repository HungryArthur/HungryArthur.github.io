Мы уже умеем передавать данные через `WithValue`, но главная сила пакета `context` — это возможность **безопасно отменять** любые долгие операции в горутинах.

Часто бывает так:
- Одна горутина выполняет тяжёлую задачу (вычисления, поиск, чтение большого файла)
- Другая горутина (или основная программа) решает, что результат уже не нужен
- Нужно быстро и чисто прервать работу, не оставляя «зависших» горутин

Для этого в `context` есть три функции, создающие **отменяемые контексты**:

#### Три способа создать отменяемый контекст

|Функция|Как создаётся|Когда отменяется|Значение Err()|
|---|---|---|---|
|**WithCancel**|`ctx, cancel := context.WithCancel(parent)`|При ручном вызове `cancel()`|`context.Canceled`|
|**WithTimeout**|`ctx, cancel := context.WithTimeout(parent, duration)`|Автоматически через указанное время|`context.DeadlineExceeded`|
|**WithDeadline**|`ctx, cancel := context.WithDeadline(parent, time.Time)`|В указанный момент времени|`context.DeadlineExceeded`|

**Важно:** все три возвращают функцию `cancel context.CancelFunc`. Её **обязательно** нужно вызвать (лучше через `defer`), иначе будет утечка горутин, потому что функции `WithCancel`, `WithTimeout`, `WithDeadline` создают **таймеры или механизмы отслеживания**, которые работают в фоне. Эти механизмы привязаны к контексту и ждут сигнала отмены, без которого продолжат работать и будут потреблять ресурсы.

#### 1. context.WithCancel — ручная отмена
```go
package main

import (
	"context"
	"fmt"
	"sync"
	"time"
)

func worker(ctx context.Context, wg *sync.WaitGroup, name string) {
	defer wg.Done() // Уменьшаем счётчик при завершении
	
	i := 1
	for {
		select {
		case <-ctx.Done():
			fmt.Printf("%s получил отмену: %v\n", name, ctx.Err())
			return
		default:
			fmt.Printf("%s работает (шаг %d)\n", name, i)
			time.Sleep(200 * time.Millisecond)
			i++
		}
	}
}

func main() {
	// Создаём контекст с отменой
	ctx, cancel := context.WithCancel(context.Background())
	defer cancel() // Важно! Гарантируем освобождение ресурсов

	// Создаём WaitGroup
	var wg sync.WaitGroup
	
	// Запускаем горутины
	wg.Add(2) // Ожидаем завершения 2 горутин
	go worker(ctx, &wg, "Горутина 1")
	go worker(ctx, &wg, "Горутина 2")

	// Даём горутинам поработать некоторое время
	time.Sleep(1 * time.Second)
	
	fmt.Println("Главная программа: отменяю всё!")
	cancel() // ← один вызов отменяет ВСЕ горутины, получившие этот контекст

	// Ждём завершения всех горутин
	wg.Wait()
	fmt.Println("Все горутины завершили работу")
}
```

#### 2. context.WithTimeout — ограничение времени выполнения
```go
package main

import (
	"context"
	"fmt"
	"sync"
	"time"
)

func worker(ctx context.Context, wg *sync.WaitGroup, name string) {
	defer wg.Done()
	
	i := 1
	for {
		select {
		case <-ctx.Done():
			fmt.Printf("%s получил таймаут: %v\n", name, ctx.Err())
			return
		default:
			fmt.Printf("%s работает (шаг %d)\n", name, i)
			time.Sleep(200 * time.Millisecond)
			i++
		}
	}
}

func main() {
	// Создаём контекст с таймаутом 2 секунды
	ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
	defer cancel() // Важно освободить ресурсы!

	var wg sync.WaitGroup
	wg.Add(2)
	
	fmt.Println("Запускаем горутины с таймаутом 2 секунды...")
	go worker(ctx, &wg, "Горутина 1")
	go worker(ctx, &wg, "Горутина 2")
	
	// Ждём завершения всех горутин
	wg.Wait()
	fmt.Println("Все горутины завершились по таймауту")
}
```

В отличие от ручного управления через `WithCancel`, где отмена происходит по явному вызову `cancel()`, использование `WithTimeout` обеспечивает автоматическое завершение работы через заданный промежуток времени (2 секунды), гарантируя, что горутины не будут работать дольше положенного, и при проверке причины остановки `ctx.Err()` возвращает `context deadline exceeded`, что чётко указывает на превышение временного лимита, а не на ручную отмену операции.

#### 3. context.WithDeadline — точное время отмены
```go
package main

import (
	"context"
	"fmt"
	"sync"
	"time"
)

func worker(ctx context.Context, wg *sync.WaitGroup, name string) {
	defer wg.Done()
	
	i := 1
	for {
		select {
		case <-ctx.Done():
			fmt.Printf("%s завершена в %s: %v\n", name, time.Now().Format("15:04:05.000"), ctx.Err())
			return
		default:
			fmt.Printf("%s работает (шаг %d) в %s\n", name, i, time.Now().Format("15:04:05.000"))
			time.Sleep(200 * time.Millisecond)
			i++
		}
	}
}

func main() {
	// Устанавливаем дедлайн: текущее время + 2 секунды
	deadline := time.Now().Add(2 * time.Second)
	ctx, cancel := context.WithDeadline(context.Background(), deadline)
	defer cancel() // Важно освободить ресурсы!

	fmt.Printf("Дедлайн установлен на: %s\n", deadline.Format("15:04:05.000"))
	fmt.Println("Текущее время:", time.Now().Format("15:04:05.000"))
	
	var wg sync.WaitGroup
	wg.Add(2)
	
	fmt.Println("\nЗапускаем горутины...")
	go worker(ctx, &wg, "Горутина 1")
	go worker(ctx, &wg, "Горутина 2")
	
	// Ждём завершения всех горутин
	wg.Wait()
	fmt.Printf("\nВсе горутины завершились в %s\n", time.Now().Format("15:04:05.000"))
}
```

В отличие от `WithTimeout`, который задаёт относительный интервал времени, `WithDeadline` использует абсолютное время — конкретный момент, к которому операция должна завершиться, что позволяет точно синхронизировать несколько независимых процессов к одному и тому же дедлайну, а не просто ограничивать их продолжительность работы с момента запуска.

#### Отмена распространяется вниз по дереву
```go
package main

import (
	"context"
	"fmt"
)

func main() {
	// Корневой контекст
	root := context.Background()

	// Ветка A - родительский контекст
	ctxA, cancelA := context.WithCancel(root)
	defer cancelA() // Гарантируем очистку!

	// Ветка A1 - потомок ctxA
	ctxA1, cancelA1 := context.WithCancel(ctxA)
	defer cancelA1() // Гарантируем очистку!

	// Ветка B - независимая от A
	ctxB, cancelB := context.WithCancel(root)
	defer cancelB() // Гарантируем очистку!

	fmt.Println("Отменяем только ветку A...")
	cancelA() // Отменяем ctxA и ctxA1, но не ctxB

	// Проверяем состояние контекстов
	fmt.Printf("ctxA отменён: %v\n", ctxA.Err())
	fmt.Printf("ctxA1 отменён: %v (наследовал от ctxA)\n", ctxA1.Err())
	fmt.Printf("ctxB отменён: %v (не зависит от ctxA)\n", ctxB.Err())

	// Демонстрация, что ctxB всё ещё жив
	select {
	case <-ctxB.Done():
		fmt.Println("ОШИБКА: ctxB не должен быть отменён!")
	default:
		fmt.Println("✓ ctxB всё ещё активен")
	}

	// Отменяем ctxB отдельно
	fmt.Println("\nТеперь отменяем ctxB...")
	cancelB()

	fmt.Printf("ctxB отменён: %v\n", ctxB.Err())
}
```
**Вывод:**
```no-highlight
Отменяем только ветку A...
ctxA отменён: context canceled
ctxA1 отменён: context canceled (наследовал от ctxA)
ctxB отменён: <nil> (не зависит от ctxA)
✓ ctxB всё ещё активен

Теперь отменяем ctxB...
ctxB отменён: context canceled
```

Как мы говорили ранее, контексты строятся в иерархическое дерево, где каждый дочерний контекст не только наследует значения, но и поведение родителя — поэтому при отмене `ctxA` автоматически отменяется и его потомок `ctxA1`, поскольку они находятся в одной ветке иерархии, тогда как `ctxB`, созданный независимо от той же корневой ноды, остаётся нетронутым, что наглядно демонстрирует принцип распространения отмены строго вниз по дереву зависимостей.

#### Итоговая таблица всех изученных контекстов

|Контекст|Отменяется?|Как отменяется|Err()|Когда использовать|
|---|---|---|---|---|
|`Background()` / `TODO()`|Нет|—|`<nil>`|Корень дерева, заглушка|
|`WithValue`|Как родитель|Как родитель|Как родитель|Передача данных между горутинами|
|`WithCancel`|Да|По вызову `cancel()`|`context.Canceled`|Нужно вручную остановить работу|
|`WithTimeout` / `WithDeadline`|Да|Автоматически по времени|`context.DeadlineExceeded`|Ограничение времени выполнения задачи|
