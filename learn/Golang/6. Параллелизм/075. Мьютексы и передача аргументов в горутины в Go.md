В прошлом шаге мы изучили запуск горутин с помощью ключевого слова `go`. Мы узнали, что программа может завершиться до выполнения всех горутин, так как главная горутина (функция `main`) не ждёт их завершения. Кроме того, горутины могут вызывать **гонку данных** (data race), когда несколько горутин одновременно изменяют общий ресурс, что приводит к непредсказуемым результатам. В этом шаге мы рассмотрим, как мьютексы (`sync.Mutex`) решают проблему гонок данных, а также как правильно передавать аргументы в горутины и избегать проблем с переменными цикла.

**Гонки данных в горутинах**
Гонка данных возникает, когда несколько горутин одновременно читают и изменяют общий ресурс без синхронизации. Это может привести к некорректным значениям.

**Пример 1: Гонка данных**
```go
package main

import (
	"fmt"
	"time"
)

func main() {
	var count int
	for i := 1; i <= 10000; i++ {
		go func() {
			count++ // Одновременное изменение count
		}()
	}
	time.Sleep(time.Millisecond * 100)
	fmt.Printf("Count: %d\n", count)
}
```
**Вывод(примерный):**
```yaml
Count: 9909
```

**Пошаговое выполнение:**
- **Шаг 1:** Запускаются 10000 горутин, каждая увеличивает `count`.
- **Шаг 2:** Без синхронизации горутины могут одновременно читать и записывать `count`, вызывая гонку данных.
- **Шаг 3:** `time.Sleep` даёт горутинам завершиться.
- **Шаг 4:** Вывод может быть произвольным и часто меньше 10000 из-за перезаписи.

**Почему это происходит?** Операция `count++` не атомарна: она включает чтение, инкремент и запись. Если две горутины выполняют её одновременно, одна может перезаписать результат другой, теряя инкремент.

**Использование мьютекса для устранения гонок данных**

**Мьютекс (sync.Mutex)** — примитив синхронизации для обеспечения взаимного исключения. Позволяет только одной горутине одновременно владеть доступом к защищаемому ресурсу.

**Основные методы:**
- **Lock()** — захватывает мьютекс. Если мьютекс уже занят, горутина блокируется до его освобождения.
- **Unlock()** — освобождает мьютекс, позволяя другой горутине его захватить.

**Принцип работы:**  
Горутина вызывает Lock() перед входом в критическую секцию и Unlock() после её завершения. Пока одна горутина удерживает мьютекс, все другие горутины, пытающиеся его захватить, будут ждать.

**Особенности:**
- Один Lock() требует одного Unlock()
- Неправильное использование может привести к deadlock

**Пример 2: Использование мьютекса**
```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	var count int
	var mu sync.Mutex

	for i := 1; i <= 10000; i++ {
		go func() {
			mu.Lock()
			count++ // Безопасное увеличение
			mu.Unlock()
		}()
	}
	time.Sleep(time.Millisecond * 100)
	fmt.Printf("Count: %d\n", count)
}
```
**Вывод(примерный):**
```makefile
Count: 10000
```

**Пример 3: Неверное использование мьютекса(дедлок):**
```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	var count int
	var mu sync.Mutex

	for i := 1; i <= 10000; i++ {
		mu.Lock()
		count++
		// не делаем разлок мьютекса
	}
	time.Sleep(time.Millisecond * 100)
	fmt.Printf("Count: %d\n", count)
}
```
**Вывод(примерный):**

**Пошаговое выполнение:**
- **Шаг 1:** Создаётся мьютекс `mu` и переменная `count`.
- **Шаг 2:** В цикле 10000 пытаемся инкрементировать переменную, каждая попытка вызывает `mu.Lock()`, увеличивает `count`,  но затем не вызывает `mu.Unlock()` и мьютекс становится залоченным.
- **Шаг 3:** Последующий вызов `mu.Lock()`, пытается залочить мьютекс, но поскольку он залочен, начинает ожидать его разблокировки и происходит дедлок(мьютекс никогда не разлочится, горутина будет всегда в ожидании).

**Использование RWMutex для устранения гонок данных**
`sync.RWMutex` — примитив синхронизации, поддерживающий два типа блокировок: для чтения (read lock) и для записи (write lock). Он позволяет множественным горутинам одновременно читать защищённый ресурс, но только одной горутине писать в него, это может ускорить работу программы, чем всегда использовать только одну блокировку(как в `sync.Mutex`).

**Основные методы:**
- `Lock()` — захватывает мьютекс для записи. Если мьютекс занят для чтения или записи, горутина блокируется.
- `Unlock()` — освобождает мьютекс после записи.
- `RLock()` — захватывает мьютекс для чтения. Множественные горутины могут одновременно удерживать read lock, если нет write lock.
- `RUnlock()` — освобождает мьютекс после чтения.

**Принцип работы:**
- Горутина вызывает `RLock()` перед чтением ресурса и `RUnlock()` после. Множественные читатели могут работать одновременно.
- Горутина вызывает `Lock()` перед записью и `Unlock()` после. Запись блокирует все чтения и другие записи.
- Один `Lock()` требует один `Unlock()`, один `RLock()` — один `RUnlock()`.
- Неправильное использование может привести к deadlock (например, вызов `Unlock()` без `Lock()`).

**Пример 4: RWMutex**
```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	var count int
	var mu sync.RWMutex

	// Запускаем 10000 горутин для записи
	for i := 1; i <= 10000; i++ {
		go func() {
			mu.Lock()
			count++ // Безопасное увеличение
			mu.Unlock()
		}()
	}
	// Запускаем 5 горутин для чтения
	for i := 1; i <= 5; i++ {
		go func(id int) {
			mu.RLock()
			fmt.Printf("Reader %d saw count: %d\n", id, count)
			mu.RUnlock()
		}(i)
	}
	time.Sleep(time.Millisecond * 100)
	fmt.Printf("Final count: %d\n", count)
}
```
**Вывод(примерный):**
```yaml
Reader 3 saw count: 9998
Reader 2 saw count: 9998
Reader 4 saw count: 9999
Reader 1 saw count: 9999
Reader 5 saw count: 9999
Final count: 10000
```

**Пошаговое выполнение:**
- **Шаг 1:** Создаётся `sync.RWMutex` `mu` и переменная `count`.
- **Шаг 2:** Запускаются 10000 горутин для записи, каждая вызывает `mu.Lock()`, увеличивает `count`, затем `mu.Unlock()`.
- **Шаг 3:** Запускаются 5 горутин для чтения, каждая вызывает `mu.RLock()`, читает `count`, затем `mu.RUnlock()`.
- **Шаг 4:** `RWMutex` позволяет множественным читателям одновременно видеть `count`, но блокирует чтение во время записи.
- **Шаг 5:** `time.Sleep` даёт горутинам завершиться.
- **Шаг 6:** Вывод: `Final count: 10000`, а строки от читателей показывают промежуточные значения `count` (например, `Reader 1 saw count: 5000`).

**Почему это работает?** `RWMutex` предотвращает гонки данных: `Lock()` обеспечивает эксклюзивный доступ для записи, а `RLock()` позволяет множественным читателям работать одновременно, если нет активной записи. Это эффективнее `sync.Mutex` в сценариях с частым чтением.

**Передача аргументов в горутины**
Аргументы в горутины передаются через параметры функции или замыкания. Неправильная передача может привести к ошибкам, особенно если переменные изменяются.

**Пример 5: Корректная передача**
```go
package main

import (
	"fmt"
	"time"
)

func task(id int) {
	fmt.Printf("Task %d executed\n", id)
}

func main() {
	for i := 1; i <= 3; i++ {
		go task(i) // Передача i как параметр
	}
	time.Sleep(time.Millisecond * 100)
}
```
**Вывод (порядок может варьироваться):**
```iecst
Task 1 executed
Task 2 executed
Task 3 executed
```

**Пояснение:** Переменная `i` передаётся как параметр, создавая локальную копию для каждой горутины.

**Проблема с переменной цикла в старых версиях Go**
До Go 1.22 переменная цикла в `for i := range` или `for i := 0; i < n; i++` была общей для всех итераций, что вызывало ошибки при её захвате в горутинах.

**Пример 6: Проблема в старых версиях (до Go 1.22):**
```go
package main

import (
	"fmt"
	"time"
)

func main() {
	for i := 1; i <= 3; i++ {
		go func() {
			fmt.Printf("Task %d executed\n", i)
		}()
	}
	time.Sleep(time.Millisecond * 100)
}
```
**Возможный вывод:** `Task 4 executed` (три раза).

**Решение (до Go 1.22):** Передавать `i` явно:
```go
for i := 1; i <= 3; i++ {
	go func(id int) {
		fmt.Printf("Task %d executed\n", id)
	}(i)
}
```
либо можно переопределять переменную `i` внутри цикла:
```go
for i := 1; i <= 3; i++ {
	i := i // go way ;)
	go func() {
		fmt.Printf("Task %d executed\n", i)
	}()
}
```

**В Go 1.22 и новее:** Переменная цикла создаётся заново для каждой итерации, устраняя проблему:
```go
for i := range 3 {
	go func() {
		fmt.Printf("Task %d executed\n", i)
	}()
}
```
**Вывод:** `Task 0 executed`, `Task 1 executed`, `Task 2 executed` (порядок может варьироваться).

**Ключевые моменты:**
- Гонки данных возникают при одновременном изменении общего ресурса без синхронизации.
- Мьютексы (`sync.Mutex,sync.RWMutex`) устраняют гонки данных, обеспечивая эксклюзивный доступ.
- Аргументы в горутины нужно передавать явно через параметры, чтобы избежать захвата изменяемых переменных.
- До Go 1.22 переменная цикла была общей, что приводило к ошибкам; в Go 1.22+ создаётся новая переменная на каждой итерации.
- `time.Sleep` используется для учебной синхронизации, но в продакшене лучше применять `sync.WaitGroup` или каналы.
