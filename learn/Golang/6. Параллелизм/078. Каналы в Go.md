В предыдущих шагах мы работали с переменными, определёнными вне горутин, например, общими счётчиками, которые обновлялись с помощью атомарных операций. Но как передавать данные напрямую между горутинами? В Go для этого используются **каналы** — безопасный и идиоматичный способ обмена данными и синхронизации. В этом шаге мы разберём **небуферизированные каналы**: как они работают, почему их операции блокирующие, что происходит при закрытии канала, и что будет, если писать в несозданный канал. Мы рассмотрим всё это через простые примеры.

#### Что такое небуферизированные каналы?
Небуферизированные каналы — это каналы в Go, которые не имеют внутренней памяти для хранения данных. Они работают как "прямая связь" между горутинами: одна горутина отправляет данные, другая их принимает. Если никто не готов принять данные, отправитель ждёт. Если никто не отправляет данные, получатель ждёт. Это делает каналы безопасным способом передачи данных без состояний гонки.
- **Создание**: `ch := make(chan int)` — создаёт небуферизированный канал для целых чисел. Запись в несозданный канал вызовет панику.
- **Отправка**: `ch <- value` — отправляет значение в канал, блокируя горутину, пока другая не примет данные.
- **Получение**: `value := <-ch` — получает значение из канала, блокируя горутину, пока данные не будут отправлены.
- **Закрытие**: `close(ch)` — закрывает канал, сигнализируя, что больше данных не будет.

#### Блокирующая природа небуферизированных каналов
Небуферизированные каналы синхронизируют горутины, потому что их операции **блокирующие**:
- **Отправка (`ch <- value`)**: Горутина блокируется, пока другая горутина не выполнит получение (`<-ch`).
- **Получение (`<-ch`)**: Горутина блокируется, пока другая горутина не отправит данные.

Это гарантирует, что отправитель и получатель "встречаются" в момент передачи данных, что делает каналы идеальными для синхронизации.

#### Пример 1: Отправка одного числа
Простая программа, где одна горутина отправляет число, а другая его получает:
```go
package main

import (
	"fmt"
)

func main() {
	ch := make(chan int) // Небуферизированный канал
	
	go func() {
		ch <- 42 // Отправка, блокируется, пока не будет получено
	}()
	
	num := <-ch // Получение, блокируется, пока не будет отправлено
	fmt.Printf("Получено: %d\n", num) // Вывод: Получено: 42
}
```
**Вывод(примерный):**
```makefile
Получено: 42
```

**Пояснение:**
- Канал `ch` небуферизированный, поэтому горутина, отправляющая 42, ждёт, пока главная горутина не выполнит получение.
- Получение в главной горутине (`<-ch`) также ждёт, пока данные не будут отправлены.
- Это обеспечивает синхронизацию: данные передаются только когда обе горутины готовы.

#### Закрытие канала
Закрытие канала (`close(ch)`) сигнализирует, что больше данных не будет. После закрытия:
- Получение из закрытого канала возвращает **нулевое значение** для типа канала (например, 0 для `chan int`) и флаг `ok=false`.
- Отправка в закрытый канал вызывает **панику** (panic).
- Цикл `for ... range` по каналу завершается, когда канал закрыт и пуст.

**Пример 2: Закрытие канала**
```go
package main

import (
	"fmt"
)

func main() {
	ch := make(chan int)
	
	go func() {
		ch <- 1
		ch <- 2
		close(ch) // Закрываем канал
	}()
	
	for num := range ch { // Читаем до закрытия
		fmt.Printf("Получено: %d\n", num)
	}
}
```
 **Вывод(примерный):**
```makefile
Получено: 1
Получено: 2
```

**Пояснение:**
- Горутина отправляет 1 и 2, затем закрывает канал.
- Цикл `for ... range` получает все значения и завершается, когда канал закрыт.
- Если попробовать получить значение после закрытия с помощью `value, ok := <-ch`, то `ok` будет `false`, а `value` — 0.

#### Писать в несозданный канал
Если канал не создан (т.е. переменная канала равна `nil`), попытка отправки или получения вызовет **панику**. Канал должен быть инициализирован с помощью `make` перед использованием.

**Пример 3: Паника при записи в несозданный канал**
```go
package main

func main() {
	var ch chan int // ch == nil
	ch <- 42       // Паника: отправка в nil-канал
}
```
**Вывод(примерный):**
```go
fatal error: all goroutines are asleep - deadlock!

goroutine 1 [chan send (nil chan)]:
main.main()
	/tmp/sandbox2916065521/prog.go:5 +0x1c
```

**Пояснение:**
- Переменная `ch` объявлена, но не инициализирована (`ch == nil`).
- Попытка отправки (`ch <- 42`) вызывает панику с сообщением вроде `panic: send on nil channel`.
- Аналогично, получение (`<-ch`) из `nil`-канала также вызовет панику.

#### Пример 4: Передача данных для подсчёта
Рассмотрим программу, где две горутины обрабатывают части массива и отправляют результаты (количество чётных чисел) в канал:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	numbers := []int{1, 2, 3, 4}
	ch := make(chan int) // Небуферизированный канал
	var wg sync.WaitGroup

	wg.Add(2)
	// Первая горутина: чётные числа в [1, 2]
	go func() {
		defer wg.Done()
		count := 0
		for _, n := range numbers[:2] {
			if n%2 == 0 {
				count++
			}
		}
		ch <- count // Отправляем 1 (2 — чётное)
	}()
	// Вторая горутина: чётные числа в [3, 4]
	go func() {
		defer wg.Done()
		count := 0
		for _, n := range numbers[2:] {
			if n%2 == 0 {
				count++
			}
		}
		ch <- count // Отправляем 1 (4 — чётное)
	}()

	go func() {
		// делаем это в отдельной горутине, чтобы не возникло дедлока
		// потому что ch <- count не выполнится, пока не будет считано в цикле for ниже
		wg.Wait()
		close(ch) // Закрываем канал
	}()

	total := 0
	for count := range ch { // Суммируем результаты
		total += count
	}
	fmt.Printf("Even: %d\n", total) // Вывод: Even: 2
}
```
**Вывод(примерный):**
```makefile
Even: 2
```

**Пояснение:**
- Две горутины считают чётные числа в своих частях массива ([1, 2] и [3, 4]).
- Каждая отправляет свой результат (1 и 1) в небуферизированный канал, блокируясь, пока главная горутина не примет данные.
- Главная горутина собирает результаты через `for ... range` и суммирует их.
- `sync.WaitGroup` ждёт завершения горутин, после чего канал закрывается.

#### Почему небуферизированные каналы?
Небуферизированные каналы полезны, когда:
- Нужно передать данные между горутинами (например, результаты вычислений).
- Требуется синхронизация: отправитель и получатель должны "встретиться".
- Хотите избежать общей памяти, чтобы предотвратить ошибки вроде состояний гонки.

#### Частые ошибки
- **Deadlock**: Если отправитель ждёт получателя, но никто не читает канал, программа зависнет. Пример:
```go
ch := make(chan int)
ch <- 42 // Deadlock: нет получателя
```
- **Паника при записи в закрытый канал**: После `close(ch)` попытка отправки вызовет панику.
- **Паника при использовании nil-канала**: Всегда инициализируйте канал с `make`.
