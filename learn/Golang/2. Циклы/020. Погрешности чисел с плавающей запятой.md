В этом шаге мы углубим знания о вещественных числах и операциях с ними. Базовые знания о вещественных числах были даны в [этом шаге](https://stepik.org/lesson/1813514/step/5?unit=1839290). Вы узнаете, что такое машинное эпсилон, как оно используется для сравнения чисел с плавающей запятой, и как можно применять другие пороги точности для практических задач.

#### Что такое машинное эпсилон?

Машинное эпсилон (ε) — это наименьшее положительное число с плавающей запятой, такое что `1.0 + ε > 1.0`, но `1.0 + ε/2 == 1.0` в арифметике с плавающей запятой. Оно характеризует точность представления чисел в компьютере, связанную с ограничениями формата IEEE 754 (используемого в Go для типа `float64`).

Числа с плавающей запятой (например, `float64` в Go) имеют ограниченную точность из-за конечного количества битов для представления мантиссы и экспоненты. Это приводит к погрешностям округления, особенно при выполнении арифметических операций, таких как сложение или умножение.

**Пример проблемы с погрешностью:**

```go
package main

import "fmt"

func main() {
	sum := 0.1 + 0.2
	expected := 0.3
	message := fmt.Sprintf("0.1 + 0.2 = %.17f, ожидается %.1f", sum, expected)
	fmt.Println(message)
}
```
**Вывод:**
```undefined
0.1 + 0.2 = 0.29999999999999999, ожидается 0.3
```

**Комментарий:** Из-за погрешности округления результат `0.1 + 0.2` не равен точно `0.3`, а составляет `0.29999999999999999`. Прямое сравнение `0.1 + 0.2 == 0.3` может вернуть `false`, что иллюстрирует проблему с точностью чисел с плавающей запятой.

#### Форматирование чисел с заданной точностью

Хотя арифметические операции с `float64` сохраняют полную точность, при выводе чисел можно использовать форматирование для отображения с заданным количеством знаков после запятой. Например, можно вывести 0.1, 0.2 и их сумму с двумя знаками после запятой.

**Пример 1: Вывод чисел с форматированием**
```go
package main

import "fmt"

func main() {
	a := 0.1
	b := 0.2
	sum := a + b
	message := fmt.Sprintf("0.1 = %.2f, 0.2 = %.2f, сумма = %.2f", a, b, sum)
	fmt.Println(message)
}
```
**Вывод:**
```ini
0.1 = 0.10, 0.2 = 0.20, сумма = 0.30
```

**Комментарий:** Форматирование с помощью `%.2f` округляет числа до двух знаков после запятой при выводе. Однако это влияет только на отображение, а не на внутреннее представление числа. Сумма `0.1 + 0.2` всё равно равна `0.29999999999999999` в памяти, и прямое сравнение `sum == 0.3` вернёт `false`.

#### Зачем нужен машинный эпсилон?

Машинное эпсилон помогает определить, насколько близко два числа с плавающей запятой могут считаться равными. Вместо прямого сравнения `a == b` используется проверка, находится ли разность `|a - b|` в пределах некоторого малого значения, связанного с машинным эпсилоном. Например, числа `a` и `b` считаются равными, если:

```go
math.Abs(a - b) < ε * math.Abs(a)
```

где `ε` — машинное эпсилон (для `float64` примерно `2.22e-16`), а `math.Abs` возвращает модуль числа.

Однако в практических задачах не всегда необходимо использовать машинное эпсилон. В зависимости от требований задачи можно выбрать больший порог точности, который считается приемлемым. Например, если вам достаточно точности до 6 знаков после запятой, вы можете использовать порог `0.000001` вместо `ε`. Это упрощает сравнение и подходит для большинства приложений, где высокая точность не критична.

Машинное эпсилон также полезно для:

- Понимания предела точности вычислений.
- Оптимизации алгоритмов, где важна погрешность (например, в численном анализе).
- Избежания ошибок в сравнении чисел с плавающей запятой.

#### Вычисление машинного эпсилона
Машинное эпсилон можно вычислить, находя наименьшее `ε`, при котором `1.0 + ε > 1.0`. Для `float64` в Go это делается с помощью цикла, уменьшающего `ε` до тех пор, пока добавление `ε/2` к 1.0 не перестанет отличаться от 1.0.

**Пример 2: Вычисление машинного эпсилона**
```go
package main

import "fmt"

func main() {
	var e float64 = 1.0
	for 1.0+e/2.0 > 1.0 {
		e /= 2.0
	}
	message := fmt.Sprintf("Машинное эпсилон: %.17e", e)
	fmt.Println(message)
}
```
**Вывод:**
```undefined
Машинное эпсилон: 2.22044604925031308e-16
```

**Комментарий:** Цикл уменьшает `e` до тех пор, пока `1.0 + e/2.0` не станет равным `1.0` из-за ограничений точности `float64`. Полученное `e` — это машинное эпсилон.

#### Использование машинного эпсилона и других порогов для сравнения

Чтобы избежать проблем с погрешностью, как в примере `0.1 + 0.2 != 0.3`, вместо прямого сравнения `a == b` используйте проверку разности с учётом машинного эпсилона или другого приемлемого порога точности.

**Пример 3: Сравнение с машинным эпсилоном**

```go
package main

import (
	"fmt"
	"math"
)

func main() {
	a := 0.1 + 0.2
	b := 0.3
	epsilon := 2.22044604925031308e-16
	isEqual := math.Abs(a-b) < epsilon*math.Abs(a)
	message := fmt.Sprintf("0.1 + 0.2 ≈ 0.3: %t", isEqual)
	fmt.Println(message)
}
```
**Вывод:**
```yaml
0.1 + 0.2 ≈ 0.3: true
```

**Комментарий:** Проверка `math.Abs(a-b) < ε*math.Abs(a)` учитывает относительную погрешность, что делает сравнение надёжным.

**Пример 4: Сравнение с приемлемым порогом**
```go
package main

import (
	"fmt"
	"math"
)

func main() {
	a := 0.1 + 0.2
	b := 0.3
	delta := 0.000001 // Приемлемый порог точности
	isEqual := math.Abs(a-b) < delta
	message := fmt.Sprintf("0.1 + 0.2 ≈ 0.3: %t (с порогом %.6f)", isEqual, delta)
	fmt.Println(message)
}
```
**Вывод:**
```yaml
0.1 + 0.2 ≈ 0.3: true (с порогом 0.000001)
```

**Комментарий:** Использование порога `0.000001` вместо машинного эпсилона упрощает сравнение и подходит для задач, где высокая точность не требуется. Порог выбирается в зависимости от требований приложения.

#### Рекомендации
- Избегайте прямого сравнения чисел с плавающей запятой (`==`).
- Используйте `math.Abs(a-b) < ε*math.Abs(a)` для сравнения с учётом машинного эпсилона, если требуется максимальная точность.
- Для практических задач выбирайте приемлемый порог точности (например, 0.000001), если высокая точность не критична.
- Понимайте ограничения формата IEEE 754, особенно при работе с малыми или большими числами.
- Для вывода чисел с заданной точностью используйте `fmt.Sprintf("%.nf", x)`, но помните, что это влияет только на отображение, а не на внутреннее представление.

**Важно!**

Различие в результатах сложения `0.1 + 0.2` (`0.30000000000000004` или `0.29999999999999999`) связано с особенностями представления чисел с плавающей точкой по стандарту IEEE 754 и применяемым режимом округления.

Поскольку десятичные дроби 0.1 и 0.2 не могут быть точно представлены в двоичной системе, их хранение в памяти компьютера приводит к неизбежной погрешности. Конкретный результат операции (округление в большую или меньшую сторону) зависит от низкоуровневых настроек процессора и математического сопроцессора, которые определяют, как обрабатывается "машинный эпсилон" — наименьшее число, которое при добавлении к 1.0 дает результат, отличный от 1.0.

Таким образом, разница в последних знаках после запятой является нормальным явлением и может варьироваться в зависимости от архитектуры компьютера, компилятора и даже версии runtime-окружения.
